## 一、模块关系图

```
┌──────────────────────────────────────────────────────────────────┐
│                         main.py                                  │
│                    （命令行交互 / 视图层）                         │
│             解析 sys.argv → 调用 service → 格式化输出             │
│             不直接读写文件，不包含任何业务判断                      │
└──────────────────────────┬───────────────────────────────────────┘
                           │  import & call
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│                      task_service.py                             │
│                    （业务逻辑 / 服务层）                           │
│                                                                  │
│   add_task(title)          → 校验 → 创建 → 保存 → 返回           │
│   get_all_tasks()          → 从 storage 读取全部任务              │
│   get_pending_tasks()      → 筛出 pending 任务                   │
│   get_done_tasks()         → 筛出 done 任务                      │
│   complete_task(task_id)   → 定位 → 改状态 → 保存 → 返回         │
│   get_statistics()         → 返回 {total, pending, done}         │
│                                                                  │
│   职责：编排模型层和存储层，向上提供语义清晰的业务接口               │
└──────────────┬────────────────────────────┬──────────────────────┘
               │  import & call             │  import & call
               ▼                            ▼
┌──────────────────────────┐   ┌──────────────────────────────────┐
│     task_model.py        │   │      task_storage.py             │
│    （数据模型 / 模型层）   │   │    （JSON 读写 / 存储层）         │
│                          │   │                                  │
│  常量：                  │   │  TASKS_FILE = "tasks.json"       │
│    TASK_STATUS_PENDING   │   │                                  │
│    TASK_STATUS_DONE      │   │  load_tasks() → list[dict]       │
│                          │   │    文件不存在 → 返回 []           │
│  create_task(id, title)  │   │    JSON 损坏  → 返回 []           │
│    → 返回标准任务字典     │   │                                  │
│                          │   │  save_tasks(tasks) → None        │
│  validate_title(title)   │   │    UTF-8 编码, ensure_ascii=False │
│    → None | 错误描述     │   │                                  │
│                          │   │  只做 I/O，不做任何业务判断        │
│  is_valid_task(task)     │   │                                  │
└──────────────────────────┘   └──────────────────────────────────┘
```

### 依赖方向

```
main ──→ task_service ──→ task_model      (数据定义)
                       ──→ task_storage    (数据持久化)
```

- `main.py` **只认识** `task_service`，不直接 import model 或 storage。
- `task_service.py` 是唯一的粘合层——它调用 model 创建数据，调用 storage 保存数据。
- `task_model.py` 与 `task_storage.py` **互相不依赖**——它们的职责正交（一个是"长什么样"，一个是"存在哪里"）。

## 二、各模块职责说明

| 模块 | 职责 | 对外接口 | 不做什么 |
|------|------|---------|---------|
| **task_model.py** (~60行) | 定义任务的数据结构、状态常量和校验规则 | `create_task()` `validate_title()` `is_valid_task()` `TASK_STATUS_PENDING/DONE` | 不读写文件，不判断业务逻辑，不处理用户输入 |
| **task_storage.py** (~40行) | 封装 JSON 文件的读写，屏蔽文件不存在、编码错误、JSON 损坏等底层异常 | `load_tasks()` `save_tasks()` | 不做数据校验，不关心任务字段含义，不创建任务 |
| **task_service.py** (~95行) | 实现添加、查询、完成等核心业务逻辑，是模型和存储之间的编排者 | `add_task()` `get_all_tasks()` `complete_task()` `get_statistics()` `get_task_by_id()` | 不直接操作用户输入（由 main 解析），不格式化输出（由 main 渲染） |
| **main.py** (~140行) | 解析命令行参数，分发到对应的处理函数，将结果以可读方式打印给用户 | （程序的 CLI 入口） | 不读写 JSON 文件，不做任务状态判断，不定义数据结构 |

## 三、重构改动对照

| 实验 1（原始 main.py） | 实验 2（重构后） | 重构手法 |
|---|---|---|
| `DATA_FILE = "tasks.json"` 在全局 | → `task_storage.TASKS_FILE` | 移入存储模块 |
| `load_tasks()` / `save_tasks()` | → `task_storage.load_tasks()` / `save_tasks()` | 提取模块 (Extract Module) |
| `get_next_id(tasks)` | → `task_service._get_next_id(tasks)` | 移入服务模块 |
| `add_task(title)` / `mark_task_done(task_id)` | → `task_service.add_task()` / `complete_task()` | 提取模块 + 重命名（更清晰） |
| `list_tasks()` | → `task_service.get_all_tasks()` | 重命名 |
| 硬编码 `"pending"` / `"done"` 字符串 | → `task_model.TASK_STATUS_PENDING/DONE` | 引入常量 |
| 无校验函数 | → `task_model.validate_title()` | 提取函数 (Extract Function) |
| `_already_done` 哨兵值标记 | → `task_service.complete_task()` 抛 `ValueError` | 用异常替代哨兵值 |
| `cmd_add()` / `cmd_list()` / `cmd_done()` | → `main.handle_add()` / `handle_list()` / `handle_done()` | 提取 + 重命名 |
| `sys.argv` 解析逻辑散落在 main() 中 | → `main.dispatch()` 统一路由 | 提取函数 |
| `print_usage()` | → 保留在 main.py（属于 UI 层） | 保留 |

## 四、重构原则

本次重构严格遵循 **行为保持型重构（Behavior-Preserving Refactoring）**，来自 Martin Fowler《重构：改善既有代码的设计》：

1. **不改变任何外部行为** — 输入相同命令，输出完全相同
2. **小步前进** — 每次只移动一个函数，移完立即验证
3. **每个函数 ≤ 25 行** — 函数短小，命名表达意图
4. **不引入新 bug** — 所有边界情况（空输入、非法编号、重复完成、文件不存在）保持一致

## 五、课后反思

### 问：为什么软件设计常常不是"多此一举"，而是为了未来少改错？

答：

把实验 1 的 176 行代码拆成 4 个文件，表面上看是"把一个文件变成了四个文件"——好像更复杂了。但实际上，这种分层设计解决的是软件工程中两个根本性问题：

**1. 可维护性 —— 改一处不影响全身**

实验 1 的 `main.py` 里，数据定义、文件读写、业务逻辑和命令行交互全部混在一起。如果要改存储方式（比如从 JSON 换成 SQLite），需要在整个 176 行文件里找所有和文件操作相关的代码，容易漏、容易错。

实验 2 中，同样的需求只需要改 `task_storage.py` 一个文件（大约 40 行），而且只要保持 `load_tasks()` 和 `save_tasks()` 的接口不变，`task_service.py` 和 `main.py` 完全不受影响。

这就是 **单一职责原则** 的直接收益：每个模块只负责一件事，改存储不会碰业务逻辑，改界面不会碰数据定义。

**2. 可理解性 —— 让新人（包括 3 个月后的自己）能快速上手**

打开一个 176 行的 `main.py`，读者需要从头读到尾才能理解整个程序的结构。打开实验 2 的项目目录，看到：
- `task_model.py` → "哦，任务是这个结构"
- `task_storage.py` → "哦，数据存在 tasks.json 里"
- `task_service.py` → "哦，能做的操作就这些"
- `main.py` → "哦，命令行入口在这里"

每个文件不超过 100 行，每个函数不超过 25 行，读代码就像翻书的目录一样清晰。

**3. 可测试性 —— 每个模块可以独立测试**

实验 1 的程序很难写自动化测试，因为所有逻辑耦合在一起。实验 2 中：
- `task_model.py` 可以独立测试（纯函数，无副作用）
- `task_storage.py` 可以独立测试（给它临时文件路径）
- `task_service.py` 可以模拟（mock）存储层来测试业务逻辑
- `main.py` 可以模拟 service 来测试命令行交互

### 核心结论

软件设计不是"多此一举"，而是软件工程的基本素养。就像盖房子要先画图纸——没人会说图纸是"浪费时间"，因为它防止的是"盖到一半发现承重墙位置不对"这种灾难。

| | 没有设计 | 有设计 |
|---|---|---|
| 加新功能 | 到处改，提心吊胆 | 只改一个模块，心中有数 |
| 改存储方案 | 全文搜索，容易漏 | 只改 storage 层 |
| 新人接手 | 需要通读全部代码 | 按模块逐层理解 |
| 出 bug | 排查范围 = 全部代码 | 排查范围 = 一个模块 |

**一句话总结：设计是"今天多花 30 分钟画结构，明天少花 3 小时改 bug"。**
