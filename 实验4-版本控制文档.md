## 五、功能一：deadline 截止日期字段

### 5.1 功能说明

为任务增加可选的 `deadline` 字段，格式 `YYYY-MM-DD`。用户可在添加任务时通过 `--deadline` 选项指定截止日期，也可不指定（默认为 None）。新增 `overdue` 命令查看已过期的待办任务。

### 5.2 修改的文件

| 文件 | 变更内容 |
|------|---------|
| `task_model.py` | 新增 `validate_deadline()` 校验函数、`create_task()` 增加 deadline 参数、`TASK_REQUIRED_FIELDS` 增加 deadline 字段 |
| `task_service.py` | `add_task()` 接受 deadline 参数、新增 `get_overdue_tasks()` 函数、统计含 overdue 计数 |
| `main.py` | `add` 命令支持 `--deadline` 选项、新增 `overdue` 命令、任务显示含截止日期 |
| `tests/` | 新增 16 个测试用例（模型 5 + 服务 8 + 存储 1 + 适配 3） |

### 5.3 使用示例

```bash
# 添加带截止日期的任务
python main.py add "交软件工程报告" --deadline 2026-06-20

# 查看过期任务
python main.py overdue
```

## 六、功能二：priority 优先级字段

### 6.1 功能说明

为任务增加 `priority` 字段，可选值：`high`（高）、`medium`（中）、`low`（低），默认 `medium`。任务列表按优先级降序排列。新增 `priority` 命令按优先级筛选。

### 6.2 修改的文件

| 文件 | 变更内容 |
|------|---------|
| `task_model.py` | 新增 `PRIORITY_LOW/MEDIUM/HIGH` 常量、`PRIORITY_ORDER` 排序映射、`validate_priority()` 校验函数、`create_task()` 增加 priority 参数 |
| `task_service.py` | `add_task()` 接受 priority 参数、`get_all_tasks()` 按优先级排序、新增 `get_tasks_by_priority()` 筛选函数 |
| `main.py` | `add` 命令支持 `--priority` 选项、新增 `priority` 命令、显示优先级图标（🔴🟡🟢） |
| `tests/` | 新增 14 个测试用例（模型 8 + 服务 5 + 存储 1） |

### 6.3 使用示例


**评审意见 #1**

> 📍 文件：`task_model.py` 第 24 行

> **问题：** `PRIORITY_ORDER` 是一个模块级可变字典。虽然目前没有被修改的风险，但如果将来某个函数无意中修改了它，会导致排序行为异常。

> def get_priority_order() -> dict:
>     return {PRIORITY_HIGH: 1, PRIORITY_MEDIUM: 2, PRIORITY_LOW: 3}
> ```
>
> **优先级：** 🟡 中等（不影响当前功能，但影响代码健壮性）

> 📍 文件：`task_service.py` `get_all_tasks()` 函数

## 八、合并冲突解决

### 8.1 冲突产生原因

两个功能分支 `feature/deadline` 和 `feature/priority` 从同一个基础（master）分支分出，各自独立开发了 4 天（模拟）。两个分支都修改了同一批核心文件：

| 文件 | feature/deadline 修改 | feature/priority 修改 | 冲突 |
|------|----------------------|----------------------|------|
| `task_model.py` | 增加 deadline 字段、validate_deadline() | 增加 priority 字段、validate_priority() | ⚡ 冲突 |
| `task_service.py` | add_task 加 deadline 参数、get_overdue_tasks() | add_task 加 priority 参数、get_tasks_by_priority() | ⚡ 冲突 |
| `main.py` | --deadline 选项、overdue 命令 | --priority 选项、priority 命令 | ⚡ 冲突 |
| `tests/conftest.py` | SAMPLE_TASKS 加 deadline | SAMPLE_TASKS 加 priority | ⚡ 冲突 |
| `tests/test_task_model.py` | deadline 测试 | priority 测试 | ⚡ 冲突 |
| `tests/test_task_service.py` | deadline 服务测试 | priority 服务测试 | ⚡ 冲突 |
| `tests/test_task_storage.py` | deadline 存储测试 | priority 存储测试 | ⚡ 冲突 |

### 8.3 冲突解决策略

对每个冲突文件采用**"保留双方功能"**的策略：

| 文件 | 冲突类型 | 解决方式 |
|------|---------|---------|
| `task_model.py` | `create_task()` 签名不同 | 合并为 `create_task(id, title, deadline=None, priority="medium")`，TASK_REQUIRED_FIELDS 同时包含两个字段 |
| `task_service.py` | `add_task()` 参数不同 | 合并为 `add_task(title, deadline=None, priority="medium")`，同时包含 `get_overdue_tasks()` 和 `get_tasks_by_priority()` |
| `main.py` | CLI 选项和命令不同 | `_parse_add_command()` 同时解析 `--deadline` 和 `--priority`，路由表同时登记 `overdue` 和 `priority` 命令 |
| `tests/conftest.py` | SAMPLE_TASKS 字段不同 | 每条示例数据同时包含 `deadline` 和 `priority` |
| `tests/test_task_model.py` | 测试类不同 | 保留双方的测试类（TestValidateDeadline + TestValidatePriority），更新 create_task 测试覆盖双字段 |
| `tests/test_task_service.py` | 测试类不同 | 保留 TestOverdueTasks + TestPriorityFiltering，add 测试覆盖双参数 |
| `tests/test_task_storage.py` | 往返测试数据不同 | 合并为统一的往返测试，跨所有字段（含 deadline + priority） |

### 8.4 冲突解决前后对比

| | 冲突解决前 | 冲突解决后 |
|------|-----------|-----------|
| 文件状态 | 7 个文件含 `<<<<<<<` / `=======` / `>>>>>>>` 标记 | 全部清理干净 |
| 测试数量 | deadline: 63 passed, priority: 61 passed | **78 passed**（含新增的双字段联合测试） |
| 功能完整性 | 各自分支独立可用 | 两个功能同时可用，一个 `add` 命令可同时指定 `--deadline` 和 `--priority` |

### 8.5 冲突解决心得

1. **先理解再动手**：不要机械地接受某一方的版本，而要理解每个冲突点的双方意图
2. **保持测试通过**：每解决一个文件的冲突就跑一次测试，确保不引入回归
3. **小的独立分支更容易合并**：如果每个分支都改了很多文件，冲突几乎不可避免。但相比把所有改动塞在一个大分支里，小分支的合并冲突仍然更容易处理
4. **Git 标记清晰**：`<<<<<<< HEAD`（当前分支/deadline）和 `>>>>>>> feature/priority` 标记让冲突一目了然

## 十、课后反思

### 问题：为什么软件工程需要"历史记录"和"责任边界"？

**答：**

#### 1. 历史记录：时间机器的价值

Git 的版本历史就像代码的"时间机器"。在本实验中，如果合并后的代码出了问题，通过 `git log` 和 `git blame` 可以精确定位到是哪个提交引入的。每个 commit 都是一个"检查点"，可以随时回退。

更重要的是，commit message 记录了**动机**——不只是"改了什么"，更是"为什么改"。3 个月后回来看 `feat: add deadline field to task model`，立刻知道这个提交的目的是什么。

#### 2. 责任边界：谁改了什么、为什么改

通过 `git blame` 可以看到每行代码是谁写的，这**不是用来追责的，而是用来沟通的**。如果我想知道 deadline 校验逻辑为什么这样设计，我知道应该去问张三。如果 priority 排序有 bug，我知道李四最了解那段代码。

在团队协作中，清晰的责任边界意味着：
- **出问题时能快速找到懂的人**，而不是群发消息问"这是谁写的"
- **代码评审有明确的被评人**，不会出现"这不是我写的"的推诿
- **每个功能的改动可追溯**，方便审计和复盘

#### 3. 本实验的体会

模拟双人开发的整个过程让我深刻体会到：

| 没有版本控制 | 有版本控制 |
|-------------|-----------|
| 改代码靠手动备份（`main.py.bak`、`main_v2.py`…） | `git branch` 隔离开发 |
| 合并时手动对比文件，容易漏改 | `git merge` 自动检测冲突范围 |
| 出 bug 不知道什么时候引入的 | `git bisect` 二分定位 |
| 代码是谁写的全靠记忆 | `git blame` 精确到行 |
| 回退只能靠 Ctrl+Z | `git revert` / `git reset` |

#### 一句话总结

> **历史记录让代码"有据可查"，责任边界让团队"各司其职"。两者结合，就是软件工程能够规模化协作的基石。**
