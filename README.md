# CampusTask — 校园任务清单

> 软件工程实验 1：从需求到最小可用产品（MVP）

---

## 一、项目简介

CampusTask 是一个命令行版的校园任务管理工具，帮助学生记录、查看和完成日常学习任务。

### 技术栈

- **语言**: Python 3
- **数据存储**: JSON 文件（`tasks.json`）
- **交互方式**: 命令行（CLI）

### 快速开始

```bash
# 添加任务
python main.py add "完成软件工程实验1"

# 查看所有任务
python main.py list

# 完成任务（编号从 list 中查看）
python main.py done 1
```

## 二、需求分析：用户故事与验收标准

下面5条用户故事是我找了两个同学聊天后总结的。他们平时作业多，常遇到的问题有：容易忘截止日期、任务多了分不清轻重、做完的任务没记录没法回头看。

### 用户故事 1：添加任务

>用户故事1：添加任务
**作为** 一个在校学生，
**我希望** 能用简单的命令快速记下一件要做的事，
**这样** 我就不怕忘掉重要的作业或实验了。

**验收标准：**

| 编号 | 验收条件 | 预期结果 |
|------|---------|---------|
| AC-1.1 | 执行 `python main.py add "任务标题"` | 任务被成功保存，并返回新任务的编号和标题 |
| AC-1.2 | 执行 `python main.py add ""`（空标题） | 程序提示"任务标题不能为空"，不创建空任务 |
| AC-1.3 | 添加任务后检查 `tasks.json` 文件 | 文件中包含该任务记录，包含编号、标题、状态(pending)、创建时间 |
| AC-1.4 | 连续添加多个任务 | 每个任务编号递增，不重复 |

---

### 用户故事 2：查看任务列表

> **作为** 一名在校学生，
> **我希望** 能够随时查看我所有的待办任务，
> **这样** 我就能清楚自己还有哪些事要做，好安排时间。

**验收标准：**

| 编号 | 验收条件 | 预期结果 |
|------|---------|---------|
| AC-2.1 | 执行 `python main.py list`（有待完成任务时） | 列出所有任务，按"待完成"和"已完成"分组显示 |
| AC-2.2 | 执行 `python main.py list`（无任务时） | 显示"暂无任务"的友好提示 |
| AC-2.3 | 列表中每个任务显示 | 任务编号、标题、创建时间 |
| AC-2.4 | 列表末尾显示 | 统计信息：总计、待完成、已完成数量 |

---

### 用户故事 3：完成任务

> **作为** 一名在校学生，
> **我希望** 能够将已完成的任务标记为"done"，
> **这样** 我能看到自己的进度，有点成就感。

**验收标准：**

| 编号 | 验收条件 | 预期结果 |
|------|---------|---------|
| AC-3.1 | 执行 `python main.py done 1`（1 号任务存在且未完成） | 任务状态变为 "done"，显示完成提示 |
| AC-3.2 | 执行 `python main.py done 1`（1 号任务已经完成） | 提示"任务已经完成，无需重复标记" |
| AC-3.3 | 执行 `python main.py done 99`（不存在的编号） | 提示"未找到编号为 99 的任务" |
| AC-3.4 | 执行 `python main.py done abc`（非法输入） | 提示"不是有效的任务编号"，给出正确示例 |

---

### 用户故事 4：数据持久化

> **作为** 一名在校学生，
> **我希望** 我添加和完成的任务能够自动保存到文件中，
> **这样** 我关掉程序再打开，任务数据不会丢。

**验收标准：**

| 编号 | 验收条件 | 预期结果 |
|------|---------|---------|
| AC-4.1 | 添加任务后关闭程序，重新运行 `list` | 之前添加的任务仍然存在 |
| AC-4.2 | 完成一个任务后检查 `tasks.json` | 该任务的状态字段已更新为 "done" |
| AC-4.3 | 首次使用（无 `tasks.json` 文件时）运行程序 | 程序自动创建 `tasks.json`，不会报错 |
| AC-4.4 | 每个任务至少包含 | id（编号）、title（标题）、status（状态）、created_at（创建时间） |

---

### 用户故事 5：查看任务创建时间

> **作为** 一名在校学生，
> **我希望** 每条任务都能显示创建时间，
> **这样** 我能判断哪些任务已经拖延太久，需要优先处理。

**验收标准：**

| 编号 | 验收条件 | 预期结果 |
|------|---------|---------|
| AC-5.1 | 执行 `python main.py list` | 每个待完成任务下方显示其创建时间 |
| AC-5.2 | 检查 `tasks.json` 中的任务记录 | 每个任务包含 `created_at` 字段，格式为 `YYYY-MM-DD HH:MM:SS` |
| AC-5.3 | 不同时间添加的任务 | 各自记录准确的创建时间，时间戳不重复 |

## 三、使用说明

### 命令参考

| 命令 | 说明 | 示例 |
|------|------|------|
| `add "标题"` | 添加一个新任务 | `python main.py add "完成实验报告"` |
| `list` | 列出所有任务 | `python main.py list` |
| `done <编号>` | 将指定任务标记为完成 | `python main.py done 3` |

### 数据结构

`tasks.json` 中每个任务包含以下字段：

```json
{
  "id": 1,
  "title": "完成实验报告",
  "status": "done",
  "created_at": "2026-06-14 10:03:22"
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | int | 任务编号，唯一且递增 |
| `title` | string | 任务标题 |
| `status` | string | 任务状态：`pending`（待完成）或 `done`（已完成） |
| `created_at` | string | 创建时间，格式 `YYYY-MM-DD HH:MM:SS` |

## 四、实验反思

### 课后反思问题

**问：为什么"会写 Python"还不等于"会做软件"？**

**答：**

"会写 Python"仅仅意味着掌握了编程语言的语法和基本库的使用，比如会用 `list`、`dict`、会写 `for` 循环、会读写文件。但"做软件"是一件系统工程，包含了很多编码之外的关键环节：

1. **需求分析是第一步，也是最难的一步。** 这次实验我先找了两个同学聊天，搞清楚他们到底烦什么，然后把那些模糊的抱怨转化成具体的、可以验证的用户故事和验收标准。这比写代码更需要沟通能力。

2. **软件是给人用的，不是给机器跑的。** 代码能跑通只是最低要求。一个好的软件需要考虑边界情况（空输入怎么办？重复操作怎么办？文件损坏怎么办？），需要给出友好的错误提示，需要让用户一看就明白怎么用——这些和 Python 语法无关。

3. **数据设计影响全局。** 选择 JSON 存储、设计每个任务包含哪些字段、如何保证编号唯一——这些都是架构决策。如果一开始数据结构设计不好，后续改动的代价很大。

4. **代码可读性 = 可维护性。** 变量命名、函数拆分、写注释、写文档，这些不是"加分项"，是软件工程的基本要求。别人看不懂的代码就是欠的技术债。

5. **软件有生命周期。** 今天的 MVP 明天可能需要增加"优先级""截止日期""分类标签"等功能。代码结构是否容易扩展，取决于一开始的设计。这超出了"会写 Python"的范畴。

**总结：** 编程语言是工具，软件工程是方法论。就像会写字不等于会写文章，会写 Python 也不等于会做软件。
