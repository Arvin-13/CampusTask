## 五、Bug 修复过程

### 5.1 实验要求

> "要求学生故意制造一个 bug，例如把'完成任务'写成删除任务，然后观察测试如何失败，再修复。"

### 5.2 Bug 描述

在 `task_service.py` 的 `complete_task()` 函数中，正确实现是：

```python
#  正确：将状态改为 done
task["status"] = TASK_STATUS_DONE
```

如果误写为：

```python
#  Bug：错误地删除了任务
tasks.remove(task)
```

**结果**："完成任务"变成了"删除任务"，用户数据丢失。

### 5.3 测试如何发现 Bug

运行 `pytest tests/ -v` 后，以下 **3 个测试**会立即失败：

#### 失败 #1：`test_complete_task_preserves_record_not_deletes_it`

```
FAILED tests/test_task_service.py::TestDataConsistency::test_complete_task_preserves_record_not_deletes_it
  AssertionError: BUG: 任务被删除了！完成任务不应删除记录
  assert None is not None
```

#### 失败 #2：`test_task_persists_after_multiple_operations`

```
FAILED tests/test_task_service.py::TestDataConsistency::test_task_persists_after_multiple_operations
  AssertionError: assert 1 == 2
  (添加了2个任务，完成1个，只剩1个——另一个被删了)
```

#### 失败 #3：`test_statistics_after_completing_all`

```
FAILED tests/test_task_service.py::TestStatistics::test_statistics_after_completing_all
  AssertionError: assert 0 == 2
  (完成了所有任务，但统计 total=0——任务全被删了)
```

### 5.4 修复方法

**一行代码的修复：**

```diff
- tasks.remove(task)                         #  Bug
+ task["status"] = TASK_STATUS_DONE          #  修复
```

修复后重新运行测试：

```
============================= 47 passed in 0.40s ==============================
```

### 5.5 经验总结

| 环节 | 说明 |
|------|------|
| **Bug 引入方式** | 一行代码的逻辑错误（remove vs status update） |
| **发现方式** | 3 个独立的测试用例同时失败，快速定位 |
| **影响范围** | 用户数据丢失——属于严重 bug |
| **修复成本** | 改一行代码，耗时 < 1 分钟 |
| **无测试的话** | 用户需要手动操作才能发现，且数据已丢失无法恢复 |

> **核心教训**：Bug 防护测试（如 `test_complete_task_preserves_record_not_deletes_it`）就像"安全带"——平时不觉得有用，一旦出事就是救命。

可通过运行 `python tests/bug_demo.py` 查看 Bug 修复过程的完整演示。

### 6.1 分层测试策略

```
┌──────────────────────────────────────────────────────┐
│  test_task_model.py  (14 用例)                       │
│  → 纯函数，无副作用，最快                              │
│  → 验证数据结构、校验逻辑、常量定义                      │
├──────────────────────────────────────────────────────┤
│  test_task_storage.py (9 用例)                       │
│  → 用 tmp_path + monkeypatch 隔离文件系统              │
│  → 验证 JSON 读写、异常安全、编码正确性                  │
├──────────────────────────────────────────────────────┤
│  test_task_service.py (24 用例)                      │
│  → 用临时文件替换真实 tasks.json                       │
│  → 验证业务逻辑完整性、边界条件、错误处理                  │
└──────────────────────────────────────────────────────┘
```

### 6.2 关键技术要点

| 技术 | 用途 | 示例 |
|------|------|------|
| `tmp_path` | 为每个测试提供独立的临时目录 | `temp_file = tmp_path / "tasks.json"` |
| `monkeypatch` | 动态替换模块属性，隔离副作用 | `monkeypatch.setattr("task_storage.TASKS_FILE", str(temp_file))` |
| `pytest.raises` | 验证异常抛出 | `with pytest.raises(ValueError, match="不能为空")` |
| `conftest.py` | 共享 fixture，复用测试数据 | `temp_tasks_file_with_data` 预填充 3 条任务 |
| 测试类组织 | 按功能分组，适合 `-k` 筛选 | `TestAddTask` / `TestCompleteTask` / `TestStatistics` |

### 6.3 测试命名规范

所有测试函数遵循 `test_<操作>_<场景>_<预期结果>` 的命名模式：

- `test_add_task_success` — 一目了然：测试添加任务，预期成功
- `test_add_empty_title_raises_error` — 明确：空标题应抛出错误
- `test_load_corrupted_json_returns_empty_list` — 清晰：损坏JSON返回空列表

## 七、课后反思

### 问题：测试是在"找老师要分"，还是在给未来的自己提供安全网？

**答：**

这个问题触及了软件工程中测试价值的本质。

#### 表面看：测试确实在"挣分"

从学生的角度看，写测试的直接动机可能是完成实验要求、获得评分。这和在课堂上回答问题获得平时分有相似之处——都是"证明我做到了"。

但这种理解太浅了。测试的真正价值远不止于此。

#### 深层看：测试是未来自己的安全网

**1. 测试是一份"永远不会过期的文档"**

`README.md` 描述的是"程序应该做什么"，但文档可能过时。测试描述的是"程序实际做什么"，而且每次运行都会自动验证。3 个月后回来看代码，注释可能已经不准了，但测试永远准——因为它在跑。

比如，`test_complete_already_done_task_raises_error` 这个测试比任何注释都更清楚地告诉我：**已完成的任务不能被再次完成**。

**2. 测试让修改不再可怕**

实验 2 做了重构——把 176 行的一坨脚本拆成 4 个模块。如果没有测试，每次移动代码都会胆战心惊："我是不是破坏了什么？"

有了测试（实验 3），重构变成了一件安全的事：改完代码 → 跑测试 → 绿了，放心。如果红了，立刻知道哪里出错。这种即时反馈是手工测试给不了的。

**3. 测试是一种"防御性思维"的训练**

写测试强迫我从"代码怎么写"转向"这段代码可能怎么出错"：

- 用户输入空标题怎么办？ → `test_add_empty_title_raises_error`
- JSON 文件损坏怎么办？ → `test_load_corrupted_json_returns_empty_list`
- 用户重复完成同一个任务怎么办？ → `test_complete_already_done_task_raises_error`

这种思维方式的转变，比掌握 pytest 语法本身更有价值。

**4. Bug 修复实验证明了测试的真正价值**

在 Bug 修复过程中，我故意把 `complete_task` 的一行代码改错。没有测试的情况下，这个 bug 可能要等到用户数据丢失了才会被发现——那时已经晚了。

有了测试，3 个用例同时报错，几秒钟就定位了问题。**测试不是在帮老师检查作业，是在帮我避免把一个有严重 bug 的程序交出去。**

#### 结论

测试当然能满足实验要求、获得分数。但如果只看到这一层，就完全误解了测试的意义。测试真正的价值是：**它是一个安全网——在你修改代码、添加功能、重构结构的时候，它告诉你"放心，原有的功能没坏"。**

| 视角 | 测试的含义 |
|------|-----------|
| **学生视角** | 完成实验，获得评分 |
| **开发者视角** | 重构安全网，修改代码的底气 |
| **未来自己的视角** | 一份永不过期的可执行文档 |
| **用户视角** | 保障软件质量的最后一道防线 |

**一句话总结：今天的测试是为了明天省下找 bug 的时间，也是为了后天能自信地说"这个程序是对的"。**

## 八、验证与确认（V&V）

根据 IEEE 1012，验证与确认是软件开发中的两项互补活动：

| | 验证（Verification） | 确认（Validation） |
|---|---|---|
| **核心问题** | "我们正确地构建了产品吗？" | "我们构建了正确的产品吗？" |
| **检查内容** | 代码是否按规格实现 | 产品是否满足用户真实需求 |
| **本实验体现** | 47 个测试验证了每个函数的行为符合设计 | 测试用例直接对应实验 1 中与真实同学访谈得出的用户故事 |

**验证活动（本实验重点）：**

- ✅ 模型层：`create_task` 返回的数据结构符合 `TASK_REQUIRED_FIELDS` 规定
- ✅ 存储层：`load_tasks` 正确处理文件不存在、JSON 损坏等异常
- ✅ 服务层：`add_task` / `complete_task` 的行为与实验 1 验收标准一致
- ✅ 所有 5 个用户故事的验收标准均有对应测试覆盖

**确认活动（与实验 1 联动）：**

- 测试用例的输入和预期输出直接来源于实验 1 中与同学访谈得到的验收标准
- 例如：AC-1.2（空标题提示"不能为空"）→ `test_add_empty_title_raises_error`
- 例如：AC-3.2（重复完成提示"已经完成"）→ `test_complete_already_done_task_raises_error`
