# 实验8 反思报告：为什么 AI Harness 工程像软件工程？

## 一、引言

本次实验为 CampusTask 构建了一个微型 AI harness，使用户能用自然语言管理任务。系统的核心是一个流水线：用户输入 → 构造 prompt → 模型推理 → 解析输出 → 安全检查 → 执行操作 → 记录日志。虽然我们使用规则驱动的 mock_model 代替了真实大模型，但整个架构设计与真实的 AI agent 系统并无本质区别。这个过程中，我深刻体会到：AI harness 工程本质上就是软件工程——只是系统中有一个组件变得"不确定"了。

## 二、哪些工作仍由人写代码完成？

在 AI harness 架构中，以下模块完全由确定性代码实现，不依赖模型：

**1. 编排逻辑（prompt_builder、parser、run_harness 主流程）**
prompt 的构造需要精确控制格式和上下文注入；模型输出的解析需要严格的 schema 验证。这些"胶水代码"决定系统的可靠性和可维护性，必须由人精确编写。

**2. 安全策略（guardrail）**
哪些操作是危险的？什么条件下需要人工确认？这些安全边界涉及价值判断和风险评估，不能交给模型自行决定。本实验中"删除所有任务"必须触发人工确认——这个规则是人定义的。

**3. 工具执行（tool_executor）**
模型只输出意图（"我想添加一个任务"），实际调用哪个函数、如何传递参数、如何处理异常，全部由确定性代码完成。模型是"大脑"，工具是"手脚"，手脚必须可靠。

**4. 观测与评测（trace_logger、eval_runner）**
日志记录和评测基准完全由代码控制。模型输出不可控，但观测体系必须可靠——否则无法判断系统是否正常工作。

## 三、哪些工作交给了模型？

在本实验中，mock_model 承担了理解用户意图的职责：从自然语言中提取操作类型（添加/查看/完成/删除）和参数（标题、截止日期、优先级）。在真实系统中，这部分由大语言模型完成。模型的优势在于处理语言的模糊性和多样性——用户说"帮我记一下"、"加一个"、"创建"都能被理解为添加任务，这是传统 if-else 难以穷举的。

## 四、当模型可能出错时，为什么传统工程实践更重要？

**需求**：当模型可能"自由发挥"时，必须通过需求文档精确界定它应该做什么、不应该做什么。模糊的需求 + 不确定的模型 = 灾难。

**测试**：模型的输出是不确定的，同一输入可能产生不同结果。因此需要 eval_cases 作为回归测试基准，用统计指标（准确率）而非二分断言来验证。这与传统软件测试的"给定输入必有确定输出"思维完全不同——我们测试的是"足够好"而非"完美"。

**日志**：当系统行为不确定时，日志从"可选的调试工具"升级为"必需的追溯机制"。每一次模型调用、每一个决策分支都必须记录在 trace.jsonl 中，以便事后复盘。trace 变成了第一公民。

**权限**：模型可能被诱导执行危险操作（本实验中的"删除所有任务"）。因此必须在模型和工具之间插入 guardrail，对高风险操作强制人工审批。这是 Human-in-the-Loop 思想的核心——模型提建议，人类做决策。

**人工审批**：软件工程中"权限最小化"原则在 AI 时代更加重要。任何可能产生不可逆副作用的操作都必须暂停并等待人类批准。这不是束缚，而是保障系统安全性的唯一手段。

## 五、总结

构建 AI harness 让我意识到：引入模型并没有减少软件工程的负担，反而增加了新的挑战。模型让系统变得更"聪明"，但也让它变得更难以预测和验证。传统软件工程中的需求分析、接口设计、测试、日志、权限控制不仅没有过时，反而因为模型的不确定性而变得更加重要。本质上，AI harness 工程 = 软件工程 + 模型行为管理。模型只是工具箱中的一个新工具，而不是工程方法的替代品。
